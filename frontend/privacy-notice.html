1. Controller Identity

2. Categories of Data

  • Enrollment face embedding — a 512-dimensional mathematical vector representation of your face, captured once during enrollment (python -m security.enroll). Not a photograph.
  • Live verification face embedding — the same kind of vector, computed per-frame during a live session to compare against the enrolled embedding.
  • Voice embedding — a mathematical vector representation of vocal characteristics, not an audio recording.
  • Session metadata — timestamps and liveness-check flags associated with the verification session.
The enrollment face embedding is written to disk (as security/enrolled_face.json) so that it can be reused across verification sessions without re-enrolling every time. This is the one exception to "nothing is persisted" — see Section 4 for exactly what that means and how long it's kept.
Aside from the enrollment embedding above, we do not collect photographs, audio recordings, or video files, and no other data is written to disk.

3. Legal Basis

  • GDPR: Explicit consent under Article 9(2)(a) for processing special category (biometric) data.
  • US / BIPA: Written consent obtained via the in-app consent modal before any data collection begins.

4. Retention & Deletion

Enrollment embedding: persisted to disk in security/enrolled_face.json until you explicitly delete it or re-enroll (which overwrites it). It is not automatically deleted at the end of a session — it is designed to persist across sessions so you don't have to re-enroll every time. To delete it, remove the file directly or ask the team operating this instance to delete it for you.

Live verification data: per-frame face embeddings computed during a session, voice embeddings, and session metadata are held in memory only for the duration of the session. When the session ends — whether by closing the browser tab, closing the app, or clicking "End Session" — this data is dereferenced and garbage-collected from memory. None of it is written to disk.

5. Third-Party Disclosure

None. All processing occurs on your device. No data is transmitted to our servers or any third party.

This statement reflects the architecture confirmed by the engineering team as of this document's last update (see consent-copy-v1.md in the repository for the underlying confirmation). If the system architecture changes — for example, if any model is moved to a cloud-hosted endpoint, or the enrollment embedding is synced anywhere off-device — this section must be revised before that change ships.

6. Your Rights

  • Right to withdraw consent — you may end a live verification session at any time, which immediately stops all in-session data collection.
  • Right to erasure — live session data is automatically deleted when a session ends, with no further action required. The persisted enrollment embedding is deleted only on explicit request or re-enrollment (see Section 4) — it is not automatically erased at session end.

7. Data Security

The enrollment embedding is stored as plaintext JSON on local disk at security/enrolled_face.json and is not encrypted at rest in this version. In-session embeddings (live face, voice) are held in RAM only and are never persisted to localStorage, IndexedDB, or the filesystem. Communication between the verification modules occurs entirely in-memory within the same session.
